# 2026-03-08 — Board ID Corrections, Calendaring Board, and Live Data Planning

## What was accomplished

**Fixed wrong board IDs for three boards.** Motions, FOIAs, and Appeals were pointing to the wrong Monday.com boards — explaining why the snapshot only showed 2 motions and 2 appeals instead of the real counts. The correct IDs were identified and updated across all config files, scripts, and documentation.

**Added Calendaring as the 19th board.** A staff member uses this board to track notices with deadlines (hearing notices, RFEs, etc.) and links them to the relevant case. Court Cases, RFEs, and Open Forms already mirror deadline data from this board. It doesn't link directly to Profiles — it connects through the case it's attached to.

**Planned the dual-database approach for live data testing.** To safely test with real Monday.com data without exposing sensitive information in the repo, a two-database strategy was designed: `data/seed.db` (fake data, committed) and `data/live.db` (real data, gitignored). A `DB_SOURCE=seed|live` environment variable will switch between them. This lets the dashboard be tested against real data shapes and volumes while keeping PII out of version control.

## Why it matters

The wrong board IDs meant the snapshot and any future sync would miss most motions, FOIAs, and appeals — a significant data gap. With all 19 boards now correctly configured, the system is ready for a complete data sync. The dual-database plan provides a safe path to validate the dashboard against real data before going live.
