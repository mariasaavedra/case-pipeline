## 2. Live Board Sync Audit (P1–P4)

Ran `npm run dev:cli -- sync --dry-run` against all 19 live Monday.com boards for the first time since the monorepo migration. Found and fixed 4 priority classes of issues.

### P1 — CLI Flag Pass-Through Bug
`dev:cli` was `npm run dev -w @case-pipeline/cli`, causing a nested npm invocation that consumed `--dry-run` and `--verbose` before they reached the script.

**Fix:** Changed `dev:cli` in `package.json` to `tsx apps/cli/src/cli.ts` (direct invocation, no nested npm).

### P2 — 14 Ambiguous `by_type: board_relation` Resolvers (`config/boards.yaml`)
Auto-generated resolvers for multi-board-relation columns defaulted to `by_type`, which is ambiguous when a board has multiple board_relation columns.

**Fix:** Replaced all 14 with specific `by_id` entries using column IDs from the sync output. Affected boards: `_na_originals_cards_notices`, `_cd_open_forms`, `court_cases`, `profiles`, `calendaring`. Also fixed:
- `profiles.e_mail_2`: `by_type: email` → `by_id: email_mm263d18`
- `profiles.last_consult_date`: `by_type: date` → `by_id: consulted_date__1`

### P3 — Calendaring Board Lost 8 Columns
Confirmed via live API fetch that 8 columns were deleted from the Calendaring board: `deadline_status`, `method`, `deadline_type_connected`, `hearing_fees`, `mh_fees_paid_on_connected`, `tp_fees_paid_on_calendaring`, `trial_fees_paid_on_calendaring`, `connect_boards`. Fee mirrors were replaced with direct date columns; `connect_boards` replaced by named board_relation links.

**Fix:** Removed all 8 stale entries from `config/boards.yaml`.

### P4 — Housekeeping
- Removed dead `fee_ks.lm_date` config entry (pointed to deleted column `date_mkwczbpg`)
- Documented `litigation.type_of_case` type change (text → dropdown, new ID: `dropdown_mm2few38`)

## 6. Bug Audit — Work Package D: Appointments N+1 Query

**Bug:** `getAppointments` fired 9 queries per appointment row (3 for snapshot, 1 for updates, 5 for full case summary). With N rows: `9N + 1` total queries. Additionally, `getClientUpdates` was called twice per profile — once directly and once inside `getClientCaseSummary`. For 20 appointments: 181 queries per page load.

**Fix:** Batch preload pattern. After the main SELECT, collect unique `profileLocalId` values and batch-fetch all enrichment data in 9 flat queries total, regardless of row count.

**New batch functions:**
| Function | File |
|---|---|
| `batchGetClientUpdates` | `libs/query/src/updates.ts` |
| `batchGetClientContracts` | `libs/query/src/contracts.ts` |
| `batchGetClientBoardItems` | `libs/query/src/board-items.ts` |
| `batchGetClientCaseSummaries` | `libs/query/src/case-summary.ts` |
| `batchGetSnapshots` (private) | `libs/query/src/appointments.ts` |

All single-profile variants unchanged — still used by the Client 360 route.

**Query budget:** 181 (worst case, 20 appts) → **9** (flat).
