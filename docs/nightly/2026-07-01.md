# Daily Update — July 1, 2026

*Written for non-technical stakeholders. Plain language, no jargon.*

## In one line

A staff member's Monday.com connection was fixed, the deployment process was rebuilt so it can never crash the server again, and the entire codebase was audited and hardened — including encrypting all database backups.

## Fernando can connect his Monday.com again

A colleague (Fernando) couldn't connect his Monday.com account to the dashboard — it kept failing with a "scope" error. The root cause was a formatting bug in how the dashboard requested permissions from Monday.com: a technical character (a colon) was being altered in transit, and Monday.com rejected the request. That was fixed and deployed. His account connects correctly now.

---

## The server crashed — and we made sure it can't happen that way again

While deploying an update, the server ran out of memory and locked up, taking the dashboard offline for a stretch. **No data was lost** — the client database sits on a separate, protected part of the disk that the crash never touched.

The cause: the server was being asked to *build* the software (a heavy, memory-hungry job) on a machine with only 1 GB of memory. That build was taking almost an hour and eventually exhausted the machine.

**The fix is a proper deployment pipeline:**

- The software is now **built automatically by GitHub** (which has ample resources) every time a change is made — it takes about one minute there.
- The server no longer builds anything. It simply **downloads the finished software and starts it** — a matter of seconds.
- A one-command deploy script (`deploy.sh`) handles the whole update and waits to confirm the server is healthy before finishing.

A deployment went from ~50 minutes with crashes to about 10 seconds, safely. The failure mode that took us down today is now structurally impossible.

---

## Every backup is now encrypted

Database backups contain sensitive information — client details and the Monday.com access tokens for each staff member. Previously, a backup file was readable by anyone who obtained it.

Now every backup is **encrypted** (industry-standard AES-256). A leaked or stolen backup file is useless without the encryption key, which is stored separately from the backups. A recovery tool was added to restore an encrypted backup when needed.

---

## A full security audit of the whole system

The entire codebase — roughly 16,000 lines across the whole application — was reviewed for security, data integrity, and robustness. The good news: **no critical vulnerabilities and no data leaks.** The database is well-protected against common attacks, secrets are handled correctly, and login is properly enforced.

The review found several worthwhile improvements, all of which were fixed and deployed today:

- **Configuration changes are now admin-only.** Previously any logged-in staff member could add or remove appointment-board settings that affect everyone's view. Now only administrators can.
- **Staff notes can no longer get stuck.** If the server restarted at the exact moment a note was being sent to Monday.com, that note could get stranded. The system now recovers any such note automatically on restart.
- **Notes are attributed to the right person.** When a note is retried after a Monday.com outage, it now posts under the staff member who wrote it, not a shared account.
- Smaller fixes: the health-check no longer reveals internal details, error responses are cleaner, and first-login records are more accurate.

---

## Account-database hardening (continued from yesterday)

The database that holds user accounts, roles, and Monday.com connections received the same safety guarantees as the main client database: automatic, versioned upgrades with a backup taken before any change, and a clean shutdown so nothing is ever left half-written. A safeguard was also added so the system can never be left without at least one administrator.
