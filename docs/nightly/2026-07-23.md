# Daily Update — July 23, 2026

*Written for non-technical stakeholders. Plain language, no jargon.*

## In one line

The home page cards are now clickable and configurable, and — more importantly —
we found and fixed the reason the system's data kept going missing: the nightly
refresh was deleting everything *before* downloading the replacement.

---

## Home page: click a card, see every case

The six cards on the home page each showed a number and a handful of examples.
Now **clicking a card opens the full list** behind that number, with a search box.

Each row also shows a column you choose — the status of the case, its deadline,
the assigned paralegal, whatever is useful to you. Previously the cards showed
which board a case came from, which on a card titled "Open Forms" only ever said
"Open Forms."

The choice works on two levels:

- **The firm default.** An administrator picks the column everyone sees.
- **Your own preference.** Anyone can pick a different column for themselves, and
  it's remembered.

Changing the firm default never overrides someone's personal choice — it only
moves the starting point for people who haven't picked one.

**Open Forms now ignores cases parked in "Send to North Pole."** They were
inflating the count with work nobody is acting on. On the current data that takes
the card from 25 down to 14.

## Two visual fixes

**Cards no longer stretch.** Each card is now as tall as its own content instead
of being padded out to match the tallest one in the row. They also adapt properly
to any monitor size.

**The SharePoint preview is now full-size.** Opening a document from a folder
holding a single file produced a preview window a couple of lines tall — unusable.
The preview was sizing itself to the file list behind it rather than to the
screen. It now fills the window as intended. The same flaw was present, unnoticed,
in every other pop-up in the app; all of them are fixed.

## The important part: the data was being deleted before it was replaced

While checking whether client emails and notes were reaching the system, we found
that they were **not** — the table that stores them was completely empty, and had
been for a week.

The cause was the design of the nightly refresh. It **deleted the entire local
database first, then started downloading** the replacement from Monday.com. That
download takes about three hours. Anything that interrupted it — a laptop going to
sleep, a network hiccup — left the system with **no data at all**, rather than
yesterday's data.

This wasn't hypothetical. It had already happened twice:

- **July 16** — the refresh died partway. Emails and notes stayed empty for a week.
- **July 23** — a network timeout lost the entire "Open Forms" board, while the
  refresh reported success for everything else.

### What we did about it

**First, we recovered the data.** A full refresh brought in **125,000 timeline
entries** — 44,000 emails, 33,000 comments, 26,000 activities, 7,000 notes,
covering 95% of all clients. The Open Forms board was restored separately.

**Then we fixed the design**, in three steps:

1. **Turned off the automatic midnight refresh.** A three-hour job that deletes
   first should not run unattended.
2. **Added a mandatory safety copy.** A refresh now snapshots the database before
   touching it, and *refuses to run* if that snapshot fails.
3. **Rebuilt the refresh so it never deletes first.** It now updates records in
   place, matching them by their Monday.com identity. An interrupted refresh
   leaves yesterday's data intact. A board that fails simply keeps what it had.

That third change fixed a second, older bug as a side effect. Every refresh used
to reassign internal ID numbers, which is why **pinned clients kept disappearing
overnight** (reported July 16). Those IDs are now stable.

---

## The refresh is also ~19× faster

Since the system no longer rebuilds from scratch, it can ask Monday.com for *only
what changed* since last time.

On our largest board: a full download takes **568 seconds**; checking for changes
takes **30**.

One deliberate limitation: a quick refresh can't notice when something is
*deleted* in Monday.com, because it only looks at what changed. So deletions are
detected by a periodic full refresh. The system is explicit about which mode it
ran in and what that mode can and cannot see.

---

## What's next

- **Turn the automatic refresh back on**, now that it's safe — likely a quick
  refresh several times a day plus one full pass overnight.
- **Emails and activities are still the slow part** (~1.5 hours). Monday.com
  doesn't flag an item as "changed" when only its email history changes, so the
  fast path can't cover it. The plan is a rolling sweep — check a portion of
  clients each run, cycling through everyone — rather than all-or-nothing.
- These changes are on `main` and take effect on the next deploy.
