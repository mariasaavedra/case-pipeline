## The dashboard is live at app.ra-codel.com

The dashboard is now running on the firm's dedicated server and accessible to the team at `https://app.ra-codel.com`. It is secured with an SSL certificate (the padlock in the browser) and Microsoft login is required to enter.

The site runs as two Docker containers — one for the API, one for the web interface — behind an nginx proxy that handles HTTPS. Both containers restart automatically if the server reboots.

---

## Only @sharma-crawford.com accounts can log in

A security restriction was added so that only accounts with a `@sharma-crawford.com` email address can access the dashboard. Anyone who logs in with a Microsoft account from a different domain is blocked at the door, before they see any data. This means the Microsoft login screen acts as a real barrier — not just a formality.

## Monday.com connection for staff members

The dashboard allows each staff member to connect their own Monday.com account so that notes and updates they post from the dashboard are attributed to them correctly.

Two issues came up during testing with a colleague:

**Redirect going to localhost instead of the live server** — when a staff member tried to connect their Monday.com account, the system was sending them back to `localhost` (the development address) instead of `app.ra-codel.com`. This was caused by an environment variable on the server that had not been updated. It was corrected and the server was restarted with the right address.

**Scope encoding bug** — Monday.com requires certain permissions to be listed in the authorization request using a specific format (`me:read`, not `me%3Aread`). A URL encoding library was automatically converting the colons, which caused Monday.com to reject the request with an "invalid scope" error. This was fixed by building the authorization URL manually to keep the colons intact. The fix was deployed to the server today.

**Remaining question — user vs. admin permissions** — A colleague who is a regular Monday.com member (not an admin) is still experiencing issues connecting their account. Monday.com may require admin-level access to authorize third-party app connections. This is the current theory and will be investigated with the Monday.com account admin.

## Server rebuild after deployment conflict

When the updated code was deployed to the server, Docker encountered a naming conflict with a leftover container from a previous run. This prevented the new containers from starting and briefly took the site offline. The old containers were removed, the new ones were started cleanly, and the site came back up within a few minutes.
