# Daily Update — June 28–29, 2026

*Written for non-technical stakeholders. Plain language, no jargon.*

## In one line

The dashboard was hardened for real-world deployment, staff can now **log in with their Microsoft account**, and real client data is live for the first time.

---

## Login is working

The dashboard's Microsoft sign-in was not completing. Clicking "Sign in with Microsoft" would redirect to Microsoft's login page, the login would succeed, and then the page would quietly return to the sign-in screen without letting anyone in.

The cause was a port conflict on the development machine — two leftover server processes were occupying the address that the login redirect was configured to use, so Microsoft was rejecting the return trip. Those processes were cleared and the server restarted on the correct address. Login works correctly now.

---

## Notes survive Monday.com outages

Previously, if someone posted a note from the dashboard while Monday.com was unreachable (outage, slow connection, API limit), the note would simply fail with an error message and be lost.

Now, when a note is posted it is saved locally first — always. If Monday.com is available, it goes through immediately. If not, it is queued and the system will keep trying in the background, automatically, until it succeeds. The staff member sees a confirmation right away either way; nothing is lost.

The retry logic is patient: it waits 1 minute before the first retry, then 2 minutes, then 4, up to a maximum of 30 minutes between attempts. If something is truly broken after many tries, it is marked for review rather than silently discarded.

---

## The server now starts and stops cleanly

When the server shuts down (for a restart, an update, or a `docker stop` command), it now finishes any in-progress requests, flushes and closes the database cleanly, and exits — without leaving the database in a half-written state.

---

## Schema updates apply automatically — no data loss

When new features require changes to the database structure, the server now applies those updates automatically on startup. Before making any change to the live database, it takes a full backup first. Client data is never at risk.

---

## Automatic daily backups

The server now takes a backup of the full client database every night at 2:30 AM. Backups are kept for 14 days and then rotated out. The backup runs while the server is live — no downtime required.

---

## Ready for Docker deployment

The dashboard is now packaged for Docker deployment — the standard way to run server software consistently on any machine:

- The API server and the web interface each run in their own container.
- Client data lives on a separate persistent volume — it survives container restarts and updates.
- A health check confirms the API is running before the web interface becomes available.

This sets up the path for deployment on the firm's own server.
